---

## 🎯 痛点

- **一个 Agent 做不好所有事：** 单个 Agent 的上下文窗口在处理策略、代码、营销研究、商业分析时很快填满
- **没有专业化：** 通用提示产生通用输出
- **Solo Founder 倦怠：** 需要一个团队，而不是另一个要管理的工具
- **知识孤岛：** 营销研究的洞察不会自动告知开发优先级

---

## ⚡ 核心功能

1. **专业化 Agent：** 每个 Agent 有独特的角色、个性和针对其领域优化的模型
2. **共享记忆：** 项目文档、目标、关键决策可供所有 Agent 访问
3. **私有上下文：** 每个 Agent 也维护自己的对话历史和领域特定的笔记
4. **单一控制面板：** 所有 Agent 通过一个 Telegram 群组可访问
5. **定时每日任务：** Agent 主动工作而不是被要求
6. **并行执行：** 多个 Agent 可以同时处理独立任务

## 📂 共享记忆结构

```
team/
├── GOALS.md              # 当前 OKR 和优先级 (所有 Agent 读取)
├── DECISIONS.md          # 关键决策日志 (只追加)
├── PROJECT_STATUS.md      # 当前项目状态 (所有 Agent 更新)
├── agents/
│   ├── milo/             # Milo 的私有上下文和笔记
│   ├── josh/             # Josh 的私有上下文
│   ├── marketing/         # Marketing agent 的研究
│   └── dev/              # Dev agent 的技术笔记
```

## 📱 Telegram 路由配置

```
## AGENTS.md — Telegram Routing

Telegram group: "Team"

Routing:
- @milo → Strategy agent
- @josh → Business agent
- @marketing → Marketing agent
- @dev → Dev agent
- @all → Broadcast to all agents
- No tag → Milo (team lead) handles by default

Each agent:
1. Reads shared GOALS.md and PROJECT_STATUS.md for context
2. Reads its own private notes
3. Processes the message
4. Responds in Telegram
5. Updates shared files if the response involves a decision or status change
```

## 💡 关键洞察

1. **个性比你想的更重要** - 给 Agent 独特的名字和沟通风格让它更自然
2. **共享记忆 + 私有上下文** - 组合至关重要
3. **正确的模型做正确的事** - 不要用昂贵的推理模型做关键词监控
4. **定时任务是飞轮** - 真正的价值来自于 Agent 主动提供洞察
5. **从 2 个开始，不是 4 个** - 先从一个负责人 + 一个专家开始
